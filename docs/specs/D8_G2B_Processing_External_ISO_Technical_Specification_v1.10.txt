3.2.

Purpose of this document

This document provides Banks and their software suppliers with the information required to
implement a functional interface to an organization, such as a Bank or Switch, which uses the
G2B Processing System.
The specification follows the recommendations of ISO 8583 (Second edition 1993-12-15,
Reference number ISO 8583:1993(E)) Financial transaction card originated messages Interchange Message Specifications. A complete specification of supported messages is
provided, covering their content, control and transaction processing requirements.
Should clarification of any point be required, please refer to the ISO Standard.

3.3.

Scope of this document

This document describes the complete functionality of the interface. Certain aspects of the
interface are in fact optional and depending on circumstances it may be acceptable to not
implement the optional components.

Message Protocol

4.1.

Overview

This chapter provides the definition of the format and content of all messages that may be
exchanged through the G2B Processing External Interface.

This section consists of descriptions of the following:
•

Message Structure—an overview of the general structure and content of the ISO 8583 messages

•

Message Types—lists the subset of ISO 8583 message types that are supported by the interface and
provides detailed definitions of the function and usage of every defined message type

•

Data Elements—gives details of which data elements are used within each of the defined message
types

Certain fields in the messages are key fields. This means that these fields are used to match
requests to responses, and reversals to originals. These key fields must be the same across an
1100/1110, 1200/1210 or 1420/1430 message pairs. Fields 11 and 12 may be the same in a
1420 reversal advice as the original 1200 request, or they may differ. Whether they are the same
or different, the original values should be placed in Field 56 (original data elements). The other
key fields 32 and 37 must be the same in a 1420 reversal advice as the matching 1100/1200
request.

4.2.4. Bit maps
The second component of an ISO 8583 message is one or two bit maps each consisting of 64
bits numbered contiguously from the left, starting with "1". Each bit signifies the presence (1) or
absence (0) in the message of the data element associated with that particular bit.
The primary bit map (bits 1-64) will always be present in all G2B Processing External Interface
messages. If the first bit within the primary bit map is set to "1", this indicates the secondary bit
map field is also present. The secondary bit map is used to index bits 65 to 128. The secondary
bit map will only be included if at least one of the data elements 65-128 is present in the
message. If the secondary bit map is present, bit 65 will always be set to “0", indicating that no
further bit maps exist.
The bit map concept eliminates transmission of message fields full of spaces or zeroes. Since the
bit map fields will always indicate which data elements are present in a message, only data
element fields required for a particular message need be sent.
4.2.5. Data elements
The third component of a message consists of a series of data elements or fields. These fields contain the
actual transaction data required by the External System and the G2B External Interface to process
transactions across the network. Messages are constructed using the bit map as an index of the data
elements that are present.

Some key aspects concerning messages are described below:
• "Requests"
Messages are either of request or advice type. A request is a real-time transaction message
generated by POS terminal. Transaction can be approved or rejected by the host;
•

"Repeats"
Whenever a repeat message is identified, that repeat message will be identical to its
original message except Message Type Identifier;

•

"Advice"
An advice describes a transaction that is already complete. Advices are usually approved,
although in some specific cases (e.g., duplicate transmission, format error etc.) host can
reject the advice;

•

"Reversals"
Reversals are always advice type messages. They contain supplementary fields which
allow reversal matching with the original transaction to be reversed.

4.3.1. Authorization messages
Authorization request is generated by POS terminal whenever it needs approval by the acquirer
to perform a card transaction. In more general terms, an authorization is an approval or guarantee
of funds given by the card issuer to the card acceptor but, depending on card processing
conditions, the acquirer can transfer liability onto itself.
"Authorisation request" is generated by POS terminal and routed from the acquirer to issuer. An
Authorisation Request (1100) requests approval authorisation for the transaction to continue and
complete. It is not intended to permit the application of this transaction to the cardholder’s
account for the purpose of issuing a bill or statement.
"Authorisation response" is generated by POS terminal and routed from the issuer to acquirer. An
Authorisation Request Response (1110) is returned in response to an Authorisation Request
(1100) to approve or deny the request.
The general sequence to interchange messages is as follows:

•

POS sends "Authorization request" message ("1100") to the host, transmitting transaction
information needed for authorization;

•

Host returns to POS an "Authorization request" response message ("1110") specifying an
action to be taken and giving key fields necessary for response identification in future
processing.

“Authorisation Advice” is generated by POS terminal and routed from the acquirer to issuer.
An Authorisation Advice (1120) advises of an authorisation carried out on behalf of the card
issuer. It is not intended to permit application of this transaction to the cardholder’s account for
the purpose of issuing a bill or statement. The External System must acknowledge each 1120
message with an 1130 message. G2B Processing acknowledges each 1120 message with an
1130.
This message should be sent via a SAF mechanism to guarantee delivery.
"Authorisation Advice response" is generated by POS terminal and routed from the issuer to
acquirer. An Authorisation Advice Response (1130) acknowledges the receipt of an 1120 message.
The External System is not permitted to reject an authorisation advice.
The general sequence to interchange messages is as follows:

•

POS sends authorization advice request message ("1120") to the host, transmitting
transaction information needed for authorization advice;

•

Host returns to POS an authorization advice request response message ("1130") specifying
in field B.39 if status check completed successfully or not.

4.3.2. Financial messages
A financial request is generated by the POS terminal whenever it needs acquirer's approval to
perform a card transaction. In more general terms, a financial transaction (or a single message)
is a transaction from the card acceptor to the card issuer containing all the necessary data
elements for authorization, posting and reconciliation.
"Financial Transaction request" is generated by POS terminal and ATM which is routed from the
acquirer to issuer. This message type is used by the acquirer to request approval for a card related
transaction. It contains the customer and transaction information needed by the issuer center or

its authorisation processor to determine the authenticity of the card, the identity of the customer,
and the details of the transaction.
"Financial request response" is generated by POS terminal and ATM which is routed from the
issuer to acquirer. This message type is used by an issuer or its authorisation processor to respond
to a 1200 financial transaction request. It provides the issuer's decision to approve or decline a
transaction. In approval responses an authorisation code will be included. For declines, the action
code may include a request to capture the card or may identify an error that caused the decline.

The general sequence to interchange messages is as follows:

•

POS terminal sends to the host a financial request message ("1200") transmitting
transaction information necessary for authorization, posting and reconciliation,

•

Host returns to the POS a financial request response message ("1210") with instructions
what action to take and giving key fields necessary for response identification in future
processing.

“Financial Transaction Advice request” is generated by POS terminal and ATM which is routed
from the acquirer to issuer. A Financial Transaction Advice (1220) advises of previously completed
financial transaction.
G2B Processing sends 1220 messages to the External System when G2B Processing stands in to
authorise a transaction for an unavailable External System. If the transaction is approved, G2B
Processing generates an 1220 message to be sent to the External System during store-andforward processing. A 1220 message may also be sent by an External System to G2B Processing.
In this case, it is regarded as a force post transaction. 1220 messages are used to complete POS
transactions which may have been initiated with an authorisation. The External System must
acknowledge each 1220 message with a 1230 message. G2B Processing acknowledges each
1220 message with a 1230 message.
“Financial Transaction Advice response” is generated by POS terminal and ATM which is routed
from the issuer to acquirer. A Financial Transaction Advice Response (1230) acknowledges the
receipt of a 1220 message.

A financial advice transaction allows billing or posting the approved transaction amount to

4.3.2.1.

After on-line transaction (authorization)

In general, financial advice messages can be sent at any time during communication (before
reconciliation). All timers and counters mentioned in the Part 1 of this document are valid. During
sending the advices, echo test messages can also be used.
There are no on-line reversals for advice messages. To cancel or change "1220" advice message,
use "return" transactions (FLD_003[1-2] must be "20").
4.3.3. Reversal messages
Reversal is generated by POS terminal whenever it needs to nullify, in whole or in part, the effect
of a previous authorization (this means that the transaction did not complete as approved) or
transaction.
Various situations may occur when creating reversals, depending on POS terminal transactional
status and whether a reversal is to be created before or after initial request response.

5.

Data Element description

5.1.

Physical format of fields

This section provides detailed descriptions of all the data elements used by the different
message types. Each data element is considered in turn in the bit map sequence

5.1.1. Numeric data
Purely numeric data will be expressed as Binary Coded Decimal (BCD). Fields must always
contain a whole number of bytes. Where necessary, the field should be padded with leading
zeroes. For example, 123 would be expressed as two hex bytes x01, x23.
5.1.2. Non-numeric data
Fields which may contain non-numeric data (types "a", "s" or "x" ) will be expressed as ASCII. Data
should be padded with trailing spaces (hex 20).
Note that composite fields in the form x + n12 are formatted as a single ASCII character followed
by BCD characters. Thus a credit amount of 50 GBP (with decimal place is 2) would be
represented as x43(‘C’) x00 x00 x00 x00 x50 x00 and a debit amount of 99,000 lira GBP (with
decimal place is 0) as x44(‘D’) x00 x00 x00 x09 x90 x00.
5.1.3. Track 2 Data
Track 2 data (Field 35) is transmitted as packed hex data. (Each single byte contains 2 track 2
characters.) The entire track data is transmitted except for the start sentinel, end sentinel and
LRC.
If the track data length (excluding start sentinel, end sentinel and LRC) is odd, a leading 0 nibble
is prefixed, to ensure a whole number of bytes.
Each track 2 character is a 4 bit value (excluding the check bit) and each 4 bit track 2 character
is coded as a single nibble. This implies that the field separator, ‘=’ or binary bit value 1101, is
coded as nibble hex value D.
Valid values for track 2 are nibbles 0 to F, but we only expect 0-9 and D to turn up for normal
payment card track 2s.
5.1.1. Variable length data

following data, followed by the actual data. The length byte does not include itself. For packed
data, the length expresses the number of digits, not the number of bytes. The data portion of

5.1.2. Message type identifier
The Message Type Identifier is treated as a Numeric Data field.
5.1.3. Bit maps
The Bit Maps are expressed as blocks of 8 bytes. The first byte codes for fields 1-8, the second
byte codes for fields 9-16, etc. Within each byte, the bits are numbered from most significant bit
(MSB) to least significant bit (LSB). So in the first byte, the MSB represents field 1, the LSB
represents Field 8. For example, hex xB6 (10110110) represents data fields 1,3,4,6 and 7.
5.1.4. PIN Data
PIN Data (Field 52) is expressed as Binary Coded Hexadecimal (BCH). This is the same as BCD,
except that the characters ‘A’ to ‘F’ are also permitted. For example, ‘193COFA8’ would be
expressed as hex x19, x3C, x0F, xA8.

Each field representing a monetary amount has a corresponding field representing the currency.
For example, Field 4 (Amount, Transaction) has the corresponding Field 49 (Currency Code,

5.1.6. Conversion Rates
The leftmost digit denotes the number of positions the decimal separator shall be moved from

5.1.8. Data annotation for Fixed TLV (and BER-TLV) element
The following annotation codes apply to PDSs in a TLV element.
• b x (n y)
A data element in this format is numeric value represented by y decimal
digits, with each digit stored on half byte (or nibble) represented as Binary Coded Decimal
(BCD). This value is stored in x bytes, right justified with leading zeroes.
• b ..x (n ..y)
A data element in this format is a numeric value represented by up to y
decimal digits, with each digit stored on half byte (or nibble) represented as Binary Coded
Decimal (BCD). This value is stored in up to x bytes, right justified with leading zeroes.
• b x (an y)
A data element in this format is an alphanumeric string of y ASCII
characters, left justified and padded with trailing hexadecimal zeroes. This string is stored
in x bytes with each ASCII character stored in one byte.
• b x (ans y)
A data element in this format is an alphanumeric string of y ASCII characters
(including special characters), left justified and padded with trailing hexadecimal zeroes.
This string is stored in x bytes with each ASCII character stored in one byte.
• b..x (an..y)
A data element in this format is an alphanumeric string of up to y ASCII
characters, left justified and padded with trailing ASCII zeroes. This string is stored in up
to x bytes with each ASCII character stored in one byte.
• b..x (ans..y) A data element in this format is an alphanumeric string of up to y ASCII
characters (including special characters), left justified and padded with trailing ASCII
zeroes. This string is stored in up to x bytes with each ASCII character stored in one byte.
• bx
A data element in this format is a binary value represented on x bytes.
• b..x; VAR
A data element in this format is a binary value represented on up to x bytes.

Usage 1: Transtain
This field is a 20-byte hash value calculated by applying a secure hash algorithm to the
transaction id and CardSecret. The TransStain cannot be reproduced or copied from one
transaction to another. If no cardholder transaction is present in the transaction, the TransStain
is calculated using a CardSecret of zeroes. This field is present in 1100 and 1200 messages if
they represent SET transactions.
Usage 2:CAVV
This field usage contains the Cardholder Authentication Verification Value (CAVV) data for VSEC
Three-Domain Secure (3-D Secure) transactions. The CAVV is a cryptographic value calculated by
the issuer’s Access Control Server (ACS) using the issuer’s encryption key and related elements.
The CAVV value is unique to the cardholder and to the transaction that was authenticated. The
acquirer transfers the ACS data to this field when preparing the VisaNet request. Visa or the issuer
verifies the CAVV to ensure that the issuer’s ACS authenticated the cardholder for the transaction
and that its contents have not been altered. This field is present in 1100 and 1200 messages if
they represent 3D Secure transactions.

5.2.43. Field 52 Personal Identification Number (PIN) Data
Format: b 8
Description: Contains a number assigned to a cardholder intended to identify that cardholder at
the point of service.
Requirements: This field is mandatory in 1100 and 1200 requests whenever the customer enters
a PIN at the point of service.
The card issuer must decrypt the PIN block received from the G2B Processing External Interface
and verify the PIN in their card level processing checks.
To maintain the security of customer accounts PINs are never sent through the system in "clear
text". PINs must always be transmitted in "encrypted" form as outlined below. All PIN
encryption/decryption functions must be performed through an acceptable hardware device.
PINS sent to G2B Processing must be encrypted either under the AWK or under TPK, depending
on interface configuration. By default AWK is used. If interface is configured to use TPK, then
there will be separate keys for each terminal (base on value of Field 41).
PINS sent from G2B Processing will be encrypted under the IWK.

6.1.2.

Message validation

When the G2B Processing External Interface receives a message containing an invalid message
field, it will log the message and return the message to the sender with a “9” prefix (i.e. a 9xxx
message will be forwarded). The Field in Error, field in the message header, will contain the value
of the first offending field that caused the message validation failure. If the message header is
invalid then the Field in Error field will contain “999”.
When G2B Processing detects a format error in the message, an entry will be created in the event
log.
6.1.3.

Network control

The status of a particular connection between the G2B Processing External Interface and an
External System may be either "up" or "down". When a session is in an "up" state all traffic to and
from the bank will be allowed. When a session is in a "down" state only logon request traffic will
be allowed. The only way that a session's status may be changed from "down" to "up" is via a

6.5.

Preauthorisation processing Message flow

Preauthorization transaction processing functionality works with 4 messages: preauthorization
("1100"), incremental authorization ("1100"), reversal ("1420"), completion ("1220").
1. POS terminal send "1100" preauthorization message and preauthorised amount in Field
4 field (amount to be blocked) and Field 48.0111=0. POS receives in response "1110"
message with Field 37 and Field 48.0111;
2. POS send "1100" incremental message with Field 37 set to the same value as received in
response of initial preauthorization, Field 48.0111=0, Field 48.0210 to sequence number,
Field 48.0211=02 and additional blocking amount in Field 4 field (without initial blocking
amount). As a result blocked amount is Field 4 from preauthorization + Field 4 from
incremental message;
3. Terminal can send more incremental messages with the same Field 37, Field 48.0111=0,
incremented sequence number in Field 48.0210 and Field 48.0211=02 and additional
blocking amount in Field 4 field;
4. POS can send full amount (on full blocked amount till now) in "1420" message with the
same Field 48.0111=0 and fields from preauthorization in Field 4 field. Field 30 field must
contain full blocked amount till now (can be greater than in initial preauthorization
message).
5. Finally POS terminal sends "1220" completion message with the same Field 37, Field
48.0111=1 and final amount to be charged from client (all authorized amount with
preauthorization + all incremental transactions. Field 22 must reflect actual completion
entry-mode or alternatively be echo of original pre-auth value.
After "1220" completion message no other correction is allowed.

7.

Cutover

7.1.

Overview

This section contains an overview of the G2B Processing External Interface functionality
connected with business days.
Reversals, describes how the G2B Processing External Interface processes reversals when cutting
over to a new business day.
End of Day, describes the G2B Processing External Interface end-of-day processing.
Settlement, describes how the G2B Processing External Interface end-of-day processing relates
to settlement.

7.2.

Reversals

The G2B Processing External Interface SAF mechanism is used to store all reversal advices.
Reversals advices are always "approved" by the G2B Processing External Interface. The following
discussion applies to a switch situation where a reversal advice is received from Bank A and
needs to be sent to Bank B.
In normal circumstances reversal advices will reach the issuer with the minimum delay. However
the issuer may be unavailable for some reason in which case the advices remain in the SAF.
If an issuer link is unavailable when cutover takes place a special situation may occur. This is
illustrated below:
•

Cash transaction acquired by Bank A is authorised by Bank B within reconciliation
date 1/1/1

•

Bank A ATM fails to dispense cash and generates a reversal.

•

G2B Processing External Interface acknowledges the reversal with a 1430
response with reconciliation date 1/1/1

•

G2B Processing External Interface is unable to deliver reversal since link to Bank
B is down

•

G2B Processing External Interface cuts over to new date 2/1/1.

•

Bank B link remains down for some hours.

•

Bank B link is re-established.

• Bank B receives reversal with previous reconciliation date 1/1/1.
If net totals are compared prior to the link being re-established there will be an unavoidable
mismatch between net totals calculated by G2B Processing External Interface and Bank B. The
difference will be the value of the reversal transactions which Bank B has not yet received.
External Systems must be aware of the fact that they may receive reversal advices belonging to
a previous reconciliation date when they log on to the network after a period of down time. In
all cases the member bank must of course acknowledge receipt of the reversal. However it is
optional whether or not the reversal is posted to the customer account, since this may already
have been done using manual procedures.

7.3.

End Of day

Once the business day has been cutover, end of day processing can be started.
First all necessary reports will be generated. Settlement will take place based on G2B Processing
that should in most cases agree with External System totals.
The structure and content of the data areas supplied in the Extract File are detailed in Appendix
B: Extract File Record Layout. The Extract File produced by G2B Processing is first created on disk,
before it is transferred to the Member Banks' Host Systems. The method of transferring this file
is an operational procedure that the G2B Processing External Interface will define. The G2B
Processing External Interface may define more than one procedure dependent upon the type(s)
of magnetic media or communication facilities physically available. The preferred method to be
used by G2B Processing is 4mm DAT.

7.4.

Settlement

Settlement will be based on G2B Processing External Interface net totals with each Member
Bank. Reconciliation differences will be resolved as soon as possible thereafter and may result
in correction entries.

8.

Security Protocol

8.1.

Overview

All PINs passed through the G2B Processing External Interface will be encrypted into a PIN block.
The format of the PIN block will be as defined in ANSI X9.8 (=ISO DP/9564/1, format 0).
When a transaction is sent from G2B Processing to the Card Issuer, the Issuer Working Key (IWK)
is used to encrypt the PIN block.
When a transaction is sent from an acquirer to G2B Processing, the Acquirer Working Key (AWK)
is used to encrypt the PIN Block.

8.2.

ANSI X9.8 PIN Block

An ANSI X9.8 PIN block consists of 16 hex digits. It is formed by generating and combining two
blocks of data.
The first block of data has the following format:
0 L P P P P P/F P/F P/F P/F P/F P/F P/F P/F F F
Where:

L = PIN Length (4-12)
P = First 4 digits of PIN
P/F = PIN digit or Filler (hex xF)
F
= Filler (hex xF)
The second block of data has the following format:
0000SSSSSSSSSSSS

Where: S = last 12 digits of card serial number (PAN), excluding the final luhn check digit.
For example, if the card number is 4511141001037601, the datablock is 0000114100103760.
The data blocks are combined by XORing operation. The XOR operation for a single bit is defined

8.3.

Zone Control Master Keys

Keys which have to be known by G2B Processing and an external system, e.g. Acquirer Working
Key, Issuer Working Key, have to be protected when they are passed between the system. The
keys are protected by encrypting them under a Zone Control Master Key (ZCMK).
The ZCMK is generated by G2B Processing. Three components are generated: A, B and C. The
three components are sent to different people at the organisation responsible for the External
System. This ensures that no single person possesses all the components required to construct
ZCMK.
8.3.1.

Issuer/Acquirer Working Keys

The Issuer Working Key is used to encrypt PIN blocks sent from G2B Processing to a Card Issuer.
The Acquirer Working Key is used to encrypt PIN blocks sent from an acquirer to G2B Processing.
In the first instance:
Each External System has 3 IWK’s (AWK’s). These are:
The live key: this is the key currently used to encrypt PIN blocks passed between G2B Processing
and the External System.
The future key: when the next key change occurs, this will become the live key
The expired key: this was the live key prior to the last key change.

8.3.4.

IWK/AWK Validation

The procedure to validate a new key (IWK or AWK) is as follows:
Use the ZCMK to decrypt the IWK (AWK)
Check the parity of the decrypted key. The parity must be odd. If is it not, a response must be
sent with an action code of 904.
Encrypt a data block consisting of 16 zeroes against the key. Take the first four digits of the result.
This is the check value.
If the calculated check value does not match the check value from the key change message, a
response must be sent with an action code of 904.

Appendix A: Card Formats

A.1. Overview
This section defines the card formats that are supported by the G2B Processing External
Interface.
The G2B Processing External Interface supports the use of track 2. Tracks 1 and 3 are not
processed by the G2B Processing External Interface. Tracks 1 and 3 may be encoded but the data
will not be passed to and from the G2B Processing External Interface.

A.2. Track 1 Format
The ISO 7813 standard defines a format for track 1 data. It is recommended that Banks that wish
to encode Track 1 use this format.

A.3. Track 2 Format
The ISO 7813 standard defines a format for track 2 data. This standard leaves room for choices
by the card issuer. Transactions utilising the G2B Processing External Interface must be originated
by cards which contain a Track 2 which conforms to the ISO 7813 standard.
